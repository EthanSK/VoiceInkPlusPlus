The opinionated path

Ethan's recommended setup

This is not a neutral list of every possible option. It is the setup Ethan actually uses to stay fast: one light programmable mouse, one direct transcription stack, and two buttons that decide where the words go.

01 Equip
REC NEXT

Put the workflow under your thumb.

Get a mouse with at least two programmable buttons. Ethan uses the Logitech G502 X LIGHTSPEED: light, comfortable, smooth over its USB receiver, and deeply configurable in Logitech G HUB.

  • Primary side buttonVoiceInk++ toggle shortcut
  • Next buttonmacOS Next Track action
Why this mouse and mapping?

G HUB is the configuration app; G502 X LIGHTSPEED is the mouse. Assign your existing VoiceInk++ shortcut to one button and the standard media action Next Track to your Next button. VoiceInk++ owns that event whenever the black recorder/transcription bar is visible—even an ineligible press becomes a safe no-op—then gives it back to media while idle.

Start with Ethan's low-friction defaults: Soniox V5 realtime, live input writing, English transcription, direct paste, and an app-specific Mode that presses Return in agent/chat apps but deliberately does not auto-send in Chrome.

Important provider details

Add your own Soniox API key, select Soniox V5, enable Real-time in each Mode, and leave Write Realtime Transcript into Input on. Ethan's current modes keep OpenAI gpt‑5.5 selected but leave enhancement off for the quickest direct-paste loop. Other transcription providers retain final-paste-only behavior.

Two buttons. Three deliberate destinations.

The main selling point

Never wait for your words to land.

Most dictation tools make “stop” mean one thing. VoiceInk++ turns destination into a one-click decision—without making you babysit the result.

Normal stop · current at delivery

End wherever the thought belongs.

Start in one app, move while talking, then stop normally. Primary owns no saved target: whichever editable input owns keyboard focus at final delivery receives the reconciled text and its Mode decides Return.

  1. Start recording in Codex.
  2. Move to Terminal while speaking.
  3. Stop normally. The transcript lands in Terminal.

Agent destinations, evidence first

Talk straight into Codex or Claude Code.

The route model is shared across desktop composers, terminals, and editor-hosted CLIs. Background delivery remains surface-specific, and VoiceInk++ fails visibly rather than claiming support that has not been physically proven.

THE HOST RULE

A CLI has no separate macOS input. If Claude Code is running in Terminal, the locked icon says Terminal. If Codex CLI is in VS Code, it says VS Code. Configure Return for that host app only after proving it on your surface; an unavailable background route fails closed without pulling the host forward.

Public, open source, deliberately personal

Build Ethan's workflow on your Mac.

VoiceInk++ currently ships as source, not a notarized public download. You need macOS 14.4 or later, Xcode, and a few minutes to grant Microphone and Accessibility permission.

  1. 1

    Clone the fork.
    Use Ethan's VoiceInkPlusPlus repository—not the upstream VoiceInk download.

  2. 2

    Build the local app.
    The Makefile creates an ad-hoc signed VoiceInk++ app without requiring a paid developer account.

  3. 3

    Grant permission and configure.
    Allow Microphone and Accessibility, pick your model, then create per-app Modes.
